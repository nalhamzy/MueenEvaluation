# Methodology — Arabic LLM Benchmark Platform

**Companion document to:** `MUEEN_EVALUATION_REPORT_EN.md`, `MUEEN_EVALUATION_LETTER_AR.md`
**Date:** 2 May 2026

This document describes the full methodology of the Arabic LLM Benchmark Platform used to evaluate Mueen AI against three commercial frontier models. It is provided as a standalone artifact for technical review.

---

## 1. Architecture

The benchmark uses a three-layer **teacher / student / judge** architecture, augmented with a fourth **human-judge** layer for validation.

```
            Article (Arabic, ~200-400 words)
                          │
                          ▼
            ┌────────────────────────────┐
            │  Layer 1 — Teacher         │
            │  Claude Opus 4.6           │
            │  Generates reference data  │
            │  for the 4 tasks           │
            └─────────────┬──────────────┘
                          │
                          ▼
            ┌────────────────────────────┐
            │  Layer 2 — Students        │
            │  4 models execute the      │
            │  same 4 tasks              │
            │   • Qwen 3.5 397B          │
            │   • DeepSeek Chat (V3)     │
            │   • Mistral Large          │
            │   • Mueen AI (browser)     │
            └─────────────┬──────────────┘
                          │
            ┌─────────────┴─────────────┐
            ▼                           ▼
   ┌────────────────────┐    ┌──────────────────────┐
   │ Layer 3a —         │    │ Layer 3b —           │
   │ Deterministic      │    │ Automated Judge      │
   │ Scoring            │    │ GPT-5.2              │
   │ • NER F1           │    │ • Summary rubric     │
   │ • NLI weighted     │    │ • Translation rubric │
   │   accuracy         │    │                      │
   └────────────────────┘    └──────────────────────┘
                          │
                          ▼
            ┌────────────────────────────┐
            │  Layer 4 — Human Judge     │
            │  Native Arabic evaluator   │
            │  Blinded (Models A-D)      │
            │  30 articles × 4 × 2 tasks │
            │  = 240 judgments           │
            └────────────────────────────┘
```

## 2. Dataset

### 2.1 Corpus

500 Arabic articles collected across 5 categories (100 per category):
- **Culture (الثقافة)** — arts, history, heritage, religion-as-culture
- **Finance (الاقتصاد والمال)** — markets, banking, business news
- **Politics (السياسة)** — regional and international politics
- **Sports (الرياضة)** — football, athletics, regional sports
- **Tech (التقنية)** — software, AI, hardware, telecommunications

Stored as `arabic_500_dataset.json`.

### 2.2 Sampling

200 articles selected for evaluation across two independent rounds:
- **Round 1:** 100 articles, `random.seed(42)`, 20 per category. File: `selected_100.json`.
- **Round 2:** 100 articles, `random.seed(99)`, 20 per category, no overlap with round 1. File: `selected_next_100.json`.

The two-round design provided a within-study stability check: scores changed by ±0.2 points or less between the two rounds for all four models, validating the methodology's reliability.

### 2.3 Article distribution in final 200-article set

| Category | n | % |
|----------|:-:|:-:|
| Culture | 40 | 20% |
| Finance | 40 | 20% |
| Politics | 42 | 21% |
| Sports | 33 | 16.5% |
| Tech | 41 | 20.5% |

(Slight variation from exact 40/category reflects deduplication and a small number of articles where one of the four models failed to produce a parseable output.)

## 3. The 4 Tasks

### 3.1 NER — Named Entity Recognition

**Input:** Arabic article body.
**Output:** JSON dict with 4 keys (PERSON, LOCATION, ORGANIZATION, MISC), each mapping to a list of distinct entity strings.
**Reference:** Generated by Claude Opus 4.6.
**Scoring:** Deterministic.

#### Scoring formula

For each entity category, compute precision and recall after Arabic normalization:
- Diacritics (ḥarakāt) removed
- Alef variants (ا، أ، إ، آ) unified
- Definite article ال stripped
- Whitespace normalized
- Case-insensitive matching

Then compute weighted F1:
```
F1_total = (PERSON × 0.35) + (LOCATION × 0.25) + (ORGANIZATION × 0.25) + (MISC × 0.15)
```

A **hallucination penalty** is applied: any entity returned by the model that does not appear (after normalization) in the article body subtracts from the entity's category score. This prevents models from inflating recall by emitting many irrelevant entities.

Final NER score is on a 0–10 scale.

### 3.2 NLI — Natural Language Inference (Fact Verification)

**Input:** Arabic article body + 4 reference claims (provided by the teacher).
**Output:** JSON list of 4 objects: `{"claim": "<claim text>", "label": "<SUPPORTED|REFUTED|NOT_ENOUGH_INFO>"}`.
**Reference:** Teacher-generated 4 claims with ground-truth labels.
**Scoring:** Deterministic, **positional**.

#### Scoring formula

The 4 claims are matched **positionally** (first claim ↔ first reference claim, etc.) — sidestepping any claim-text rewriting by the student model. For each claim, the predicted label is compared to the ground-truth label, with weights:

```
NOT_ENOUGH_INFO  weight = 1.5   (hardest, weighted highest)
REFUTED          weight = 1.2
SUPPORTED        weight = 1.0
```

```
NLI_score = sum(correct_label_weights) / sum(reference_label_weights) × 10
```

This rewards models that correctly identify the harder labels (especially NOT_ENOUGH_INFO, which is the most semantically demanding).

### 3.3 Summary — Arabic Summarization

**Input:** Arabic article body.
**Output:** A 2-sentence Arabic summary in formal Modern Standard Arabic (فصحى).
**Reference:** Teacher-generated 2-sentence summary.
**Scoring:** GPT-5.2 LLM-as-Judge.

#### Judge rubric (0–10 scale, derived from a 0–11 raw score)

| Criterion | Range | Description |
|-----------|:-----:|-------------|
| Factual accuracy | 0–3 | Every claim in the summary is supported by the article |
| Coverage | 0–3 | Captures the main events / facts of the article |
| No added inference | 0–2 | No information in summary that isn't in the source |
| Register fluency | 0–2 | Formal MSA, no dialect, grammatical |
| Verbatim copying penalty | 0–1 | Penalty for copy-paste from article (subtracts from total) |

Raw total (0–11) is rescaled to 0–10. The judge is given the article, the reference summary, and the model's summary, and is asked to apply the rubric to the model's summary using the article as the ground truth (the reference is provided as one example, not as a gold standard the model must match).

### 3.4 Translation — English → Arabic

**Input:** A 3-sentence English summary of the article (provided as part of the dataset).
**Output:** Arabic translation in formal MSA.
**Reference:** Teacher-generated Arabic translation.
**Scoring:** GPT-5.2 LLM-as-Judge.

#### Judge rubric (0–10 scale)

| Criterion | Range | Description |
|-----------|:-----:|-------------|
| Faithfulness | 0–3 | Translation says exactly what the English says |
| Fluency | 0–3 | Reads as natural, formal Arabic |
| Terminology | 0–2 | Proper nouns, technical terms, numbers handled correctly |
| Register | 0–2 | Formal MSA appropriate to the source register |

Raw total (0–10) is the final score.

## 4. Overall Score

The four task scores are combined into a single overall score:

```
Overall = NER × 0.30 + NLI × 0.20 + Summary × 0.25 + Translation × 0.25
```

### Choice of weights

The weights reflect the user's stated priorities for an Arabic NLP system:
- NER (0.30) — the most discriminating deterministic signal across the 4 models
- Generative tasks combined (0.50) — Summary + Translation, the user-facing capabilities
- NLI (0.20) — saturated for commercial models (all > 9.4), included as a baseline check

The weights are not derived from a formal optimization. **The model ranking is robust to any reasonable reweighting** because Mueen AI ranks last on every individual component; the only quantity that changes under reweighting is the *gap size*.

## 5. Statistical Methodology

### 5.1 Bootstrap confidence intervals

All point estimates in the report are accompanied by **95% non-parametric bootstrap confidence intervals**:

- **Resampling:** 1000 iterations of resampling articles with replacement
- **Statistic:** mean of the per-article task scores within each model
- **CI:** 2.5th and 97.5th percentiles of the resampled means
- **Seed:** `random.Random(11)` for reproducibility

A pairwise model difference is reported as **statistically significant at 95%** when the two models' CIs do not overlap.

### 5.2 Spearman correlation (human vs. LLM judge)

Used in Section 8 of the report. Computed at three levels:
- **Per-judgment:** n = 240 (30 articles × 4 models × 2 tasks). Tests whether the human and LLM judges agree on individual quality assessments.
- **Per-task per-judgment:** n = 120 each for Summary and Translation. Tests whether agreement varies by task.
- **Model-level:** n = 4 (the 4 model means). Tests whether the human and LLM judges produce the same model ranking.

Spearman is preferred over Pearson because the underlying scales (1–5 for human, 0–10 for LLM judge) are not commensurable in their absolute values, but rank order is meaningful in both.

## 6. Infrastructure

### 6.1 Backend stack

- **Language:** Python 3.12
- **Framework:** FastAPI
- **Database:** SQLite with WAL mode (`backend/data/db.sqlite`)
- **ORM:** SQLAlchemy 2.0
- **Schema:** Pydantic v2

### 6.2 Frontend stack

- **Framework:** Angular 20 standalone components
- **UI:** Angular Material
- **Charts:** Chart.js
- **Styling:** SCSS

### 6.3 LLM provider integrations

A single multi-provider LLM client (`backend/services/llm_client.py`) abstracts the differences between providers:

| Provider | API | Auth | Used for |
|----------|-----|------|----------|
| Anthropic | Native + Batch | API key | Teacher (Claude Opus 4.6) |
| OpenAI | Native | API key | Judge (GPT-5.2) |
| AWS Bedrock | Converse API | AWS SigV4 | Mistral Large |
| Dashscope | Native | API key | Qwen 3.5 |
| DeepSeek | Native | API key | DeepSeek Chat |
| (manual) | Browser | N/A | Mueen AI |

### 6.4 Cost optimization

- **Anthropic Batch API** used for teacher dataset generation (50% cost discount).
- **OpenAI Batch API** attempted for the judge — fell back to synchronous calls when GPT-5.2 was not yet available in the batch endpoint at evaluation time.

### 6.5 Determinism and reproducibility

- All sampling uses explicit seeds (`random.seed(42)`, `random.seed(99)`, etc.)
- Bootstrap CI computation uses seed=11
- Human evaluation builder uses seed=7
- All seeds are documented in the relevant artifact files

---

## 7. Quality Assurance

- **53+ unit tests** in `backend/tests/` covering scoring, teacher, student, judge, and routing logic.
- **Mocked LLM calls** in tests prevent flakiness and reduce CI cost.
- **Manual spot-checks** were performed on Mueen's outputs for each round; the first NLI round was re-run after a prompt fix once it became clear the model was rewriting claims rather than copying them.

## 9. Reproduction Recipe

```bash
# 1. Start the platform (one-time)
py -m uvicorn backend.main:app --port 8111
cd frontend && npx ng serve --port 4222

# 2. Upload the corpus
curl -X POST http://localhost:8111/api/articles/upload \
  -F "file=@arabic_500_dataset.json"

# 3. Sample articles for evaluation (deterministic via seeds)
py scripts/sample_100.py --seed 42 --per-category 20    # round 1
py scripts/sample_100.py --seed 99 --per-category 20    # round 2

# 4. Generate teacher dataset and run student evaluations
py scripts/run_benchmark.py --per-category 20 --seed 42
py scripts/run_benchmark.py --per-category 20 --seed 99

# 5. Compute bootstrap confidence intervals
py scripts/compute_bootstrap_ci.py --iters 1000 --seed 11

# 6. Build the human evaluation package (blinded workbook)
py scripts/build_human_eval_package.py --seed 7 --per-category 6

# 7. After human evaluator returns the filled workbook:
py scripts/ingest_human_eval.py --filled human_eval/human_eval_workbook_Ahmed.xlsx

# 8. Build the deliverable report
py scripts/build_mueen_report.py
```

All steps are deterministic given the seeds.
